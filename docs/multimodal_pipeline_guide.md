# 多模态客服智能体 - 技术流程详解

## 一、核心问题：图片到底是怎么"嵌入"的？

**先说结论：在我们当前的系统中，图片并不是直接被转成向量的。**

这是一个常见的误解。我们的系统采用的是 **"图片 → 文字描述 → 文字向量"** 的间接路线，而不是像 CLIP 那样直接把图片像素编码成向量。下面解释为什么以及怎么做的。

### 1.1 为什么不直接嵌入图片？

直接嵌入图片需要用到专门的视觉编码器（如 CLIP、SigLIP），这类模型会把一张图片编码为一个固定维度的向量，然后就可以和文本向量在同一个空间里做相似度比较。

但这有几个实际问题：
- CLIP 的向量空间和我们的文本嵌入模型（BGE）的向量空间**不兼容**，不能直接混在同一个向量库里检索
- 说明书中的图片（流程图、接线图、面板示意图）包含大量**文字标注和结构信息**，CLIP 对这类图片的理解不如多模态大模型
- 赛题评分强调的是"图文互补"，关键是**找对图**而不是做图像相似度搜索

### 1.2 我们实际的做法：Caption 方案

```
┌──────────┐     多模态 LLM      ┌──────────────┐    BGE 模型     ┌──────────┐
│  图片文件  │ ──────────────────→ │ 文字描述(Caption) │ ─────────────→ │ 文本向量  │
│  .png/.jpg │   "描述这张图片"    │  300字以内     │   text→vector   │ 768维     │
└──────────┘                     └──────────────┘                 └──────────┘
```

具体流程（对应代码 `src/preprocess/caption_images.py`）：

1. 把图片文件读取为 **base64 编码**字符串
2. 构造一个包含图片和提示词的消息，发送给**多模态大模型**（如 GPT-4o、Gemini）
3. 大模型返回一段 **中文文字描述**（如："图片展示的是洗衣机控制面板，从左到右依次为电源键、模式选择旋钮、水位调节按钮..."）
4. 把这段文字描述用 **BGE 文本嵌入模型** 转成 768 维向量
5. 这个向量就可以和其他文本片段的向量一起存到 Milvus 里了

**核心思想：让多模态大模型充当"翻译官"，把图片翻译成文字，然后一切回到纯文本 RAG 的流程。**

### 1.3 各种输入类型的嵌入方式对比

| 输入类型 | 处理方式 | 最终嵌入的内容 |
|---------|---------|---------------|
| 纯文本（TXT） | 直接切分 | 文本片段 → BGE → 向量 |
| PDF 文档 | PyPDFLoader 按页提取文字 | 每页文字 → 切分 → BGE → 向量 |
| DOCX 文档 | Docx2txtLoader 提取文字 | 文字内容 → 切分 → BGE → 向量 |
| PDF 中的图片 | PyMuPDF 提取图片 → 多模态 LLM 生成 Caption | Caption 文字 → BGE → 向量 |
| 独立图片文件 | 同上，多模态 LLM 生成 Caption | Caption 文字 → BGE → 向量 |
| 非结构化表格 | PDF/DOCX 解析时包含在文字中 | 表格文字 → 切分 → BGE → 向量 |

> 注意：**无论什么类型的输入，最终被嵌入的都是"文字"**，只是获取文字的方式不同。

## 二、向量存在哪里？Milvus 的存储架构

### 2.1 我们用的是 Zilliz Cloud（云端托管的 Milvus）

当前配置（`configs/config.yaml`）：

```yaml
vector_store:
  connection_args:
    uri: "https://in03-xxx.serverless.gcp-us-west1.cloud.zilliz.com"
    token: "${ZILLIZ_TOKEN}"
  collection_name: "multimodal_customer_service_v1"
```

**是的，向量保存在云端。** 具体来说：

- **Zilliz Cloud** 是 Milvus 的官方托管服务（类似于 MongoDB Atlas 之于 MongoDB）
- 数据存储在 GCP（Google Cloud Platform）的 US-West1 区域
- 通过 HTTPS API 进行读写，本地不存储向量数据
- Serverless 模式，按用量计费

### 2.2 向量库里每条记录长什么样？

```
┌─────────────────────────────────────────────────────────────────┐
│  Milvus Collection: multimodal_customer_service_v1              │
├──────────┬────────────┬──────────┬────────────┬────────────────┤
│ id (自动) │ text       │ vector   │ source     │ related_images │
├──────────┼────────────┼──────────┼────────────┼────────────────┤
│ 1        │ "本产品支持 │ [0.12,   │ "说明书.pdf"│ ["p3_img1.png"]│
│          │  三种洗涤…" │  -0.34,  │            │                │
│          │            │  ...]    │            │                │
├──────────┼────────────┼──────────┼────────────┼────────────────┤
│ 2        │ "控制面板从 │ [0.08,   │ "说明书.pdf"│ ["p5_img1.png",│
│          │  左到右…"   │  0.21,   │            │  "p5_img2.png"]│
│          │            │  ...]    │            │                │
├──────────┼────────────┼──────────┼────────────┼────────────────┤
│ 3        │ "图片展示了 │ [-0.15,  │ "p3_img1"  │ ["p3_img1.png"]│
│  (Caption)│ 洗衣机面板" │  0.42,   │ (图片来源)  │                │
│          │            │  ...]    │            │                │
└──────────┴────────────┴──────────┴────────────┴────────────────┘
```

说明：
- **前两行** 是从说明书中切分出来的文本片段
- **第三行** 是图片 Caption 生成的文字描述
- 它们使用同一个 BGE 模型编码，向量维度相同，所以能在同一个集合中检索

## 三、用户提问时上传图片，系统怎么处理？

这是整个系统中最关键的多模态交互环节。流程如下：

```
用户输入                          系统处理
─────────                        ────────
"我的洗衣机出现      ┌───────────────────────────────────┐
 E3 错误代码，  ────→│ 步骤1：检测到有图片，调用多模态 LLM │
 怎么办？"           │ 把图片转为文字描述                  │
 + [一张照片]        └──────────────┬────────────────────┘
                                   ↓
                    ┌───────────────────────────────────┐
                    │ 步骤2：拼接用户文字 + 图片描述       │
                    │                                   │
                    │ "我的洗衣机出现E3错误代码，怎么办？  │
                    │                                   │
                    │  【用户上传图片描述】：图片显示洗衣机  │
                    │  LED面板上闪烁E3代码，排水指示灯亮红  │
                    │  色，门锁指示灯正常。"               │
                    └──────────────┬────────────────────┘
                                   ↓
                    ┌───────────────────────────────────┐
                    │ 步骤3：用拼接后的文字去向量库检索     │
                    │ BGE编码 → Milvus 相似度搜索         │
                    │ → 命中说明书中关于 E3 错误码的段落    │
                    │ → 同时带出该段落关联的配图路径        │
                    └──────────────┬────────────────────┘
                                   ↓
                    ┌───────────────────────────────────┐
                    │ 步骤4：将检索到的文字+配图路径       │
                    │ 一起喂给大模型，生成最终回答          │
                    └───────────────────────────────────┘
```

对应代码调用链：

```
agent.answer(question="...", image_path="photo.jpg")
  │
  ├→ multimodal_input.build_multimodal_query()     # 图片→文字描述，拼接到query
  │     └→ describe_image()                        # 调用多模态LLM
  │           └→ encode_image_to_base64()           # 图片→base64
  │
  ├→ chain_of_thought.decompose_question()          # 拆解子问题（如有）
  │
  ├→ multimodal_retriever.retrieve(query)           # 多模态检索
  │     ├→ text_retriever.invoke(query)             # 文本向量检索+Rerank
  │     └→ image_retriever.get_images_for_docs()    # 从metadata取关联图片
  │
  ├→ hallucination.apply_hallucination_filter()     # 幻觉抑制
  │
  └→ chain_of_thought.merge_answers()               # 合并子回答
```

**关键点：用户上传的图片不会被直接嵌入向量库做搜索，而是被多模态 LLM "翻译"成文字后，走的仍然是文本检索通道。**

## 四、图片和文档位置是怎么关联的？

这是整个系统的"桥梁"设计，分为**预处理阶段的关联**和**检索阶段的传递**两部分。

### 4.1 预处理阶段：建立图文映射

```
                    说明书.pdf
                 ┌─────────────┐
                 │  第 1 页      │
                 │  文字内容...  │
                 │              │
                 │  第 2 页      │
                 │  文字内容...  │
                 │  [图片A]     │──→ 提取图片A，记录：来自第2页
                 │              │
                 │  第 3 页      │
                 │  文字内容...  │
                 │  [图片B]     │──→ 提取图片B，记录：来自第3页
                 │  [图片C]     │──→ 提取图片C，记录：来自第3页
                 └─────────────┘
```

**步骤一：双路径解析同一份 PDF**

| 解析工具 | 提取什么 | 关键元数据 |
|---------|---------|-----------|
| **PyPDFLoader**（LangChain） | 每页的文字内容 → Document 对象 | `metadata.page = 0, 1, 2...`（0 起始） |
| **PyMuPDF**（fitz） | 每页的内嵌图片 → 图片文件 | `page = 1, 2, 3...`（1 起始） |

两条路径独立工作，但都记录了**页码**。

**步骤二：通过页码建立映射**（对应代码 `parse_manual.py → build_image_text_mapping()`）

```python
# 图片记录：(source_pdf="说明书.pdf", page=3)
# 文本记录：(source="说明书.pdf", page=2)  ← 注意 PyPDFLoader 是 0-indexed
# 匹配逻辑：text.page + 1 == image.page  → 同一页！

key = (source, page + 1)  # 对齐两套页码
if key in page_to_images:
    doc.metadata["related_images"] = [图片路径列表]
```

**步骤三：写入映射文件**

最终生成 `data_processed/image_text_mapping.json`：

```json
{
  "说明书.pdf_page2": {
    "source": "说明书.pdf",
    "page": 2,
    "text_preview": "本产品控制面板从左到右依次为...",
    "images": ["说明书_p3_img1.png", "说明书_p3_img2.png"]
  }
}
```

同时，每个 Document 对象的 metadata 里也注入了 `related_images` 字段。这意味着当这个文本片段被切分、嵌入、存入 Milvus 后，**图片路径作为元数据一起保存了**。

### 4.2 检索阶段：图片如何跟着文本一起返回

```
用户提问 "E3 错误码怎么处理？"
           │
           ↓
   ┌─── Milvus 向量检索 ───┐
   │  命中文本片段:          │
   │  "E3错误码表示排水异常" │
   │  metadata:             │
   │    source: 说明书.pdf   │
   │    page: 15            │
   │    related_images:     │──→ 【关联图片路径自动带出】
   │      - p16_img1.png    │
   │      - p16_img2.png    │
   └────────────────────────┘
           │
           ↓
   ┌─── format_docs() ───────────────────────────────┐
   │  [片段1 - 来源: 说明书.pdf - 得分 0.87]:        │
   │  E3错误码表示排水异常，请检查排水管...            │
   │                                                 │
   │  【相关配图路径】：                               │
   │  data_processed/images_extracted/p16_img1.png    │
   │  data_processed/images_extracted/p16_img2.png    │
   └─────────────────────────────────────────────────┘
           │
           ↓
      大模型基于文字+图片路径生成回答
      （前端可根据路径展示图片）
```

如果文本检索命中的段落**没有直接关联图片**（比如该页没有图），系统还有一个兜底机制：

```
命中的文本段落没有 related_images
           │
           ↓
   image_retriever.search_captions(query)
   在图片 Caption 中搜索关键词
           │
           ↓
   如果找到相关 Caption → 补充其对应的图片路径到返回结果中
```

### 4.3 完整数据流总览

```
 ┌────────── 预处理阶段（离线，运行一次）──────────────────────────────────┐

```

## 五、诚实说明：当前方案的局限性和可能的优化

### 5.1 当前方案的局限

| 局限 | 说明 |
|------|------|
| **图片关联粒度粗** | 以"页"为单位关联，同一页有多张图，全部关联到该页所有文本段落 |
| **Caption 质量依赖 LLM** | 如果多模态 LLM 对某张图的描述不准确，后续的检索和回答都会受影响 |
| **不支持图-图检索** | 用户上传图片后，是翻译成文字再检索，不能直接做图片相似度搜索 |
| **非结构化表格解析弱** | PDF 中的复杂表格用 PyPDFLoader 提取后可能丢失格式 |

### 5.2 可以优化的方向

1. **引入 CLIP 做图-图检索**：为图片额外生成 CLIP 向量，存到独立的 Milvus Collection 中，支持用户上传图片直接搜索相似图片
2. **用 pdfplumber 优化表格提取**：`pdfplumber` 可以识别 PDF 中的表格结构并转为 Markdown
3. **细化图文关联**：使用 PDF 中图片的 bbox（边界框）坐标，和文字的位置做更精确的匹配，而不是仅靠页码
4. **微调多模态模型**：用说明书图片+人工标注的 Caption 数据对模型做微调，提升图片描述的准确性

## 六、对外讲解时的一句话总结

> 我们的多模态 RAG 系统**不直接嵌入图片像素**，而是用多模态大模型把图片"翻译"成文字描述后，统一走文本向量检索通道。图片与文档的关联通过 PDF 页码映射实现——预处理时从同一页分别提取文字和图片，绑定它们的对应关系，检索时文字命中则自动带出关联配图。
